> How does it work                                                              
● Five layers, each doing different work:

  ---
  1. Rules — always on

  Markdown files in ~/.claude/rules/ecc/. Load every session. Define what "good"   means: test-first, conventional commits, no hardcoded secrets, 80% coverage.
  No prompting required.

  ---
  2. Skills — context-activated knowledge

  When you work on Django, django-patterns activates. When you write tests,
  tdd-workflow activates. Skills inject domain expertise automatically — you
  don't invoke them, they appear when relevant.

  ---
  3. Agents — isolated specialists

  You ask /code-review. Claude Code spawns the code-reviewer agent in a child
  context with limited tools (Read, Grep, Glob — no write access). It reads your
   changed files, returns findings. Your main session only receives the
  conclusion, not 20 tool calls.

  Your session → delegate → agent (isolated context) → result → your session

  ---
  4. Hooks — event automations

  Every tool event can trigger a script:

  You edit a .ts file
    → PostToolUse hook fires
    → TypeScript typecheck runs
    → Error appears immediately (not at build time)

  You run a bash command
    → PreToolUse hook fires
    → Scans for sk-, ghp_, AKIA patterns
    → Blocks if secret detected

  Session ends
    → Stop hook fires
    → Patterns extracted → stored as instincts

  ---
  5. Continuous learning — improves over time

  Each session, the observe hook records what worked. /evolve clusters instincts
   into new skills. Next session, your own patterns are active alongside ECC's
  built-in ones.

  ---
  The full picture

  Session starts
    ↓ SessionStart hook injects project memory
    ↓ Rules load (always)
    ↓ Skills activate based on context

  You work
    ↓ Hooks enforce standards automatically
    ↓ Agents handle delegated tasks in isolation
    ↓ Skills provide domain knowledge on demand

  Session ends
    ↓ Stop hook saves state + extracts patterns
    ↓ Next session picks up where you left off

  Deep dive: Onboarding (docs/getting-started/onboarding.md) — walks through a
  real feature implementation with all five layers in action.

> How do skills,agents and skills work together                                 
● Skills, agents, and hooks play different roles — but they're designed to
  complement each other:

  ---
  The mental model

  Rules       → what good looks like (always on)
  Skills      → knowledge: "here's how to do this" (passive, injected)
  Agents      → execution: "go do this bounded task" (active, isolated)
  Hooks       → enforcement: "this runs automatically" (event-triggered)

  ---
  How they combine in practice

  Skills feed agents.
  When you delegate to the code-reviewer agent, the security-review skill is
  active in its context. The agent doesn't just review generically — it has
  ECC's OWASP checklist, pattern library, and decision criteria injected. The
  skill is the knowledge; the agent is the executor.

  Hooks invoke the workflow, agents do the work.
  The post:edit:typecheck hook fires after every file edit — but it's a fast
  script, not an LLM. When you need deeper analysis (/code-review), that's an
  agent. Hooks handle the mechanical; agents handle the intelligent.

  Skills + hooks = consistent behavior without prompting.
  The tdd-workflow skill tells Claude Code how to do TDD. The
  post:edit:typecheck hook enforces it after every edit. Together: you never
  have to say "remember to write tests first" or "run the typecheck."

  ---
  A real example: fixing a bug

  1. You describe the bug

  Why separate them?

  ┌─────────┬─────────────────────┬────────────────────┬────────────────────┐
  │         │       Skills        │       Agents       │       Hooks        │
  ├─────────┼─────────────────────┼────────────────────┼────────────────────┤
  │ Speed   │ Instant (injected   │ Slower (LLM call)  │ Fast (script,      │
  │         │ text)               │                    │ <200ms)            │
  ├─────────┼─────────────────────┼────────────────────┼────────────────────┤
  │ Cost    │ Near-zero           │ Token cost         │ Near-zero          │
  ├─────────┼─────────────────────┼────────────────────┼────────────────────┤
  │ Scope   │ Your whole session  │ Isolated child     │ One tool event     │
  │         │                     │ context            │                    │
  ├─────────┼─────────────────────┼────────────────────┼────────────────────┤
  │ Purpose │ Know how            │ Do it              │ Enforce it         │
  └─────────┴─────────────────────┴────────────────────┴────────────────────┘

  Skills are cheap to keep active. Agents are expensive — use them for bounded
  tasks that need isolation. Hooks are unconditional — use them for things that
  must always happen.

  The design principle: put knowledge in skills, execution in agents,
  enforcement in hooks.
